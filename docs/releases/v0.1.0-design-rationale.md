# v0.1.0 Design Rationale

This release freezes a small, deterministic provenance kernel for developer and
auditor use.

Normative protocol behavior is specified in `PROTOCOL.md`. This note documents
engineering rationale for the `v0.1.0` architecture decisions.

## 1) Why The Core Is Small

This release freezes the core operational set as:

- `put`
- `use`
- `xform`
- `attest`
- `revoke`
- `alias_claim`

with `trace` as the deterministic projection/query surface.

The protocol intentionally avoids adding extra core verbs in `v0.1.x` to keep
conformance simple, replay stable, and cross-language parity auditable.

## 2) Why The Layers Are Separated

The protocol distinguishes:

- identity/signature layer
- replay/order layer (`(t, mh)`)
- lineage/projection layer (`trace`)
- annotation layer (`attest`, `revoke`)
- naming layer (`alias_claim`)

This separation is the main reason the system stays deterministic and auditable
across implementations.

## 3) What This Means For Future Work

Future work should prefer expansion around the core, not expansion of the
core.

Preferred directions:

- spaces
- federation
- transport
- wallet-like identity workflows

The core protocol verb set should remain stable unless a strict correctness gap
requires change.
